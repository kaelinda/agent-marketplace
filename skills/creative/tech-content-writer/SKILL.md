# 技术文章写作 SKILL (tech-content-writer)

## 定位

为「AICoder」技术公众号提供标准化写作流程。产出目标：**无 AI 味、由浅入深、简洁严谨、优先引用官方文档**的技术长文。

## ⭐ 用户最常见的硬要求

按用户反馈频率排序（2026-06 月统计）：

1. **「段落化」+「写得太短」**（2026-06-15 用户原话"太短了，并且没有段落"）→ 见「段落级要求」一节
2. **「改 20:9 横向封面」**（2026-06-14 用户硬要求）→ 见 `wechat-cover-image` skill
3. **「带图发布」+「用 md-to-html 渲染」**（2026-06-18 用户硬要求）→ 见 `tech-wechat-publish` skill
4. **「同素材写两版」**（用户重复要求"收录"同一推文时）→ 见 Pitfalls 节"v1 + v2 强制差异化"

最常被忽略的：**段落化**和**字数是反向约束**，必须双满足。

## 核心写作原则

### 0. 写之前先做读者价值评估（用户硬约束，2026-06-29）

技术文章动笔前必须先判断：这篇内容对目标读者是否真的有帮助。不要直接把官方文档、API 列表、会议纪要或代码片段堆成文章。

写作前先回答 4 个问题：

1. **读者是谁**：iOS 开发者、AI 工具用户、工程负责人，还是泛技术读者？
2. **读者卡在哪里**：接入决策、踩坑排查、架构取舍、版本迁移、性能优化，还是只是知道有这个 API？
3. **这篇文章给什么增量价值**：判断框架、避坑清单、最小可运行路径、对比表、实战经验、发布前 checklist。
4. **为什么读者不直接看官方文档**：如果答案只是“中文翻译”，这篇文章不合格；必须提供官方文档之外的工程判断。

合格文章应优先产出：
- 决策框架：什么时候该用、什么时候别用
- 常见坑：第一天会踩什么、上线前会出什么问题
- 最小闭环：读者 10 分钟能跑通的路径
- 实战判断：作者作为开发者会怎么选
- 读者收益：读完能少走哪一步弯路

不合格信号：
- 只是罗列 API
- 只是复述 WWDC / README / 博客
- 代码很多但没有解释为什么这么写
- 只有“能做什么”，没有“什么时候该做、怎么做才稳”
- 没有读者场景、没有坑、没有取舍

### 1. 由浅入深 (Progressive Complexity)

```
概念铺垫 → 最小示例 → 逐步深入 → 边界/陷阱 → 实战总结
```

- 开篇用一句话说清「这个东西是什么、解决什么问题」
- 先给最小可运行示例（MRE），再展开原理
- 每个新概念引入时，先给直觉解释，再给精确定义
- 不要上来就甩架构图 + 20 个术语

### 2. 简洁严谨 (Concise & Rigorous)

- 一句话能说清的不用两句
- 技术表述必须准确：不模糊、不臆测、不含「大概」「可能」「应该」
- 数据/性能声明必须标注来源或测量条件
- 代码示例必须可运行（或明确标注「伪代码」）

### 3. 优先引用官方文档 (Official Docs First)

- **每个关键技术点必须附官方文档链接**
- 优先级：官方文档 > GitHub README > 技术博客 > Stack Overflow
- 引用格式：`[文档标题](url)` 或行内 `官方文档指出：...`
- 不引用过时资料（标注版本/日期）
- 如果官方文档有中英文版本，优先引用英文原版（更准确），括号附中文版链接

**引用模板：**
```
根据 [Hermes Agent 官方文档](https://hermes-agent.nousresearch.com/docs/tools#terminal)，
terminal 工具支持 background 模式……
```

### 4. 绝对禁用句式

以下句式 **在任何上下文中都不允许出现**：

| 禁用句式 | 替代方案 |
|---------|---------|
| `不是 X，而是 Y` | 直接说 Y 是什么，或用「本质上是 Y」「核心在于 Y」 |
| `与其说 X，不如说 Y` | 直接陈述 Y |
| `与其 X，不如 Y` | 直接陈述 Y |
| `与其 X，不如说 Y` | 直接陈述 Y |
| `与其说是 X，不如说是 Y` | 直接陈述 Y |

**示例：**
```
❌ 错误：Agent 不是简单的脚本，而是一个具备自主决策能力的系统。
✅ 正确：Agent 本质上是一个具备自主决策能力的系统。

❌ 错误：与其说 Prompt 是指令，不如说它是一种编程语言。
✅ 正确：Prompt 本质上是一种面向 LLM 的编程语言。
```

## 文章结构模板

### 段落级要求（用户硬约束，2026-06-15 实测）⭐

用户说「**没有段落**」/「**段落化**」/「**写得太密了**」时，按以下规则生成：

- **每段 2-5 句话，80-200 字**（含 h3 标题的话 100-200 字符）
- **每段独立 `### h3` 小标题**（关键章节可缩到 2 句话一段 + 标题）
- 段与段空行隔开（`\n\n`），不要 4-5 段挤在一起
- 列表用 `-` 或 `1.`，**每条 bullet 独立短句**，不写长复合句
- 资源区每条链接要带**一行简介**，不裸链接堆叠

**「段落化 + 甜点字数」是反向约束**（2026-06-15 三版迭代总结）：
| 段长中位数 | 段数 | 总字数 | 命中区间 |
|---|---|---|---|
| 50-80 字 | 35-45 段 | 1500-2000 字 | ❌ 太短 |
| 100-200 字 | 30-40 段 | 3000-4000 字 | ✅ 甜点区 |
| 250-500 字 | 12-20 段 | 3500-4500 字 | ⚠️ 大段平铺（用户会嫌"没段落"） |
| 600+ 字 | 5-10 段 | 3000-4000 字 | ❌ 散文化 |

**扩写策略**（当段已密但字数不够时）：
1. 每段多说一层「为什么是这样」+「我第一次遇到这事的场景」
2. 在合适位置**新增独立小段**（如「verifier 维护的三个现实负担」独立成 3 段）
3. 资源区扩成 6 个独立小段（每段：标题 + 一行简介 + 链接）
4. 加「附录：N 种反模式」+ 每条独立 h3 段 + 一段总结

**当用户同时说"段落化 + 短/不够"时**（2026-06-15 用户原话「太短了，并且没有段落」）：**双约束同时满足**。DeepSeek 容易陷入"加 h3 但每段 < 50 字 → 字数 1500"的陷阱，必须显式在 prompt 里写"每段 100-200 字 + 至少 25 个 h3 标题 + 总字数 3000-4000"三个指标都满足。

**DeepSeek 第一次生成的常见缺陷**（2026-06-15 三版实测）：
- 倾向"段太长 + 段数少"：3500 字 / 18 段 / 段长 400+ → 命中"散文化"
- 倾向"段太碎 + 字数少"：1800 字 / 50 段 / 段长 40 → 命中"太短"
- 解决：prompt 中**必须给段数 + 段长 + 总字数 三个具体数字**，让模型知道目标

### 模板 A：工具/框架介绍（最常用）

```
1. Hook（100字）— 一句话说清是什么 + 解决什么痛点
2. 最小示例（300字）— 5 分钟上手的 Hello World
3. 核心概念（800-1200字）— 3-5 个关键概念，每个配代码片段
4. 实战场景（600-800字）— 1-2 个真实工程场景
5. 踩坑与边界（300-500字）— 常见错误 + 官方文档没提到的
6. 总结 + 资源（200字）— 速查表 / 官方链接 / 下一步
```

### 模板 B：问题排查 / 踩坑记录

```
1. 问题描述（100字）— 报错信息 + 环境
2. 排查过程（500字）— 逐步缩小范围，展示思考链路
3. 根因分析（300字）— 为什么出错（引用源码/文档）
4. 解决方案（300字）— 代码 diff 或配置变更
5. 举一反三（200字）— 同类问题的通用排查思路
```

### 模板 C：原理深挖

```
1. 直觉解释（200字）— 用类比让读者建立心智模型
2. 最小实现（500字）— 从零造一个玩具版
3. 源码/论文解读（800-1200字）— 逐层拆解
4. 对比（300字）— 与同类方案的差异表
5. 工程启示（200字）— 对实际开发的影响
```

### 模板 D：探索型（从单一信息源扩展为科普/解读文章）

当源材料是「一条推文」「一篇官方文档」「一个 GitHub README」「一段截屏 OCR」等**单一信息点**，用户说「收录/探索/整理成技术博客」时，不是模板 A 的工具介绍、也不是模板 B 的踩坑记录，而是**探索型**：先讲故事背景，再展开主体。

```
1. 信息源故事（300字）— 作者是谁、为什么这条内容重要、附带截图 OCR / 原文关键句
2. 用户痛点（200字）— 这条内容触发的开发者日常工作痛点
3. 官方/原作者权威解读（600-1200字）— 从 primary source（lldb.llvm.org、github README、官方 docs）抓关键章节，逐节解读
4. 实操配置步骤（500-800字）— 让读者 5 分钟能在自己环境复现的最小路径
5. 与之前方案的对比（400字）— 替代方案对比表、迁移成本、收益
6. 工程意义延伸（300字）— 拉远视角，看这件事对所在领域（iOS / CI / 前端）的长期影响
```

**判别方法**：用户原话包含「收录」「探索」「整理」「thread 内容」+ 信息源是 1 条推文 / 1 个文档 / 1 个仓库 → 探索型。

**关键差异**（vs 模板 A）：
- 模板 A 开篇直接给「这是 X、解决 Y」定义；模板 D 开篇**讲故事**（谁发的、内容是什么、有多少人看）
- 模板 A 第三节是「核心概念」（教读者懂）；模板 D 第三节是「**官方/原作者权威解读**」（替读者读文档）
- 模板 A 第四节是「实战场景」（开发者怎么用）；模板 D 第四节是「**实操配置步骤**」（最少 5 分钟能跑通）

**避免**：把模板 D 硬套模板 A 的「Hook → 最小示例 → 核心概念 → 实战 → 总结」节奏，会丢失探索型文章的「先讲故事再展开」的天然节奏，读起来像在背书。

**触发词**（2026-06-25 实测高频）：
- 「收录 + 整理成技术文章」
- 「探索这篇文章提到的内容」
- 「根据推文中 thread 的内容」
- 「X (推文链接) + 整理 + 发布到公众号」

### 模板 D 的灵魂是「决策框架」，不是「API 列表」（用户 2026-06-29 强纠偏）

用户对探索型文章的真实期望**不是「把作者原话翻译一遍」**。原话：

> 「这篇文章不合格，作为技术文章，并不能吸引读者，或者并不会给读者带来价值，请重写」

翻译/复述 = 失败。合格探索型文章必须能回答读者心里的一个问题：

> 「读完这篇，我下一次遇到类似情况要做什么不同的决定？」

落到模板 D 的 6 段，**重心从「第三、四节」移到「第一节 + 最后一节」**：

| 段落 | 旧重心（失败） | 新重心（合格） |
|------|----------------|----------------|
| 信息源故事 | 介绍作者发了什么 | **提取作者没说但隐含的判断**：他为什么强调「不要让 agent 当设计师」？这反映了他踩过什么坑？ |
| 用户痛点 | 转述背景 | **指出读者在自己项目里也会遇到的同样判断**：哪个 API 选型、哪种实现顺序、哪种取舍？ |
| 官方解读 | 按章节复述 transcript | **只摘那些「能改变读者决策」的关键句**，把解释性铺垫删掉 |
| 实操步骤 | 演示配置 | **给一个最小可跑通的最小判断**，而不是 5 段代码 |
| 与之前方案对比 | 列替代品 | **告诉读者「如果你现在用的是 X，可以立刻改成 Y」** |
| 工程意义延伸 | 总结全文 | **给出一个 1 句话的「决策判断」**（如"先广撒网、再 hybrid、再做 tuning panel"） |

**质量自检（发布前问自己）**：
- [ ] 文章去掉所有「作者说」「WWDC 提到」这种转述句，是否还成立？
- [ ] 读者读完能不能立刻在他自己的项目里做一个不同的决策？
- [ ] 这篇文章如果明天发布，**没有任何官方文档更新的情况下**，还成立吗？
- [ ] 是否给了至少 1 个「反常识」判断（作者在 transcript 里没说但其实暗示了的）？

如果任何一项是「否」，重写。重写时优先保留「工程意义延伸」和「用户痛点」两节，中间章节全部重写。

### 探索型文章的最小可用内容集

实测下来一篇合格探索型文章只需要：

1. **推文/原帖 OCR + 作者背景 1 段**（50-100 字）
2. **3-5 条「决策框架」**（每条 150-200 字，独立 h3）
3. **一段可执行代码或 checklist**（80-150 行）
4. **一段反向观点或避坑清单**（150-200 字）
5. **一段 1 句话收束**（50 字以内）

总长 1500-2000 字中文叙述就足够。**禁止堆砌 3500+ 字**——那是模板 A 的标准，探索型不需要。

## 系列文章规范（Agent 工程化系列）

### 系列标识

- 每篇标题格式：`Agent 工程化系列 Vol.XX | {具体标题}`
- `alt_titles` 提供 3 个备选标题（数字+反常识+身份代入公式）
- 正文末尾收束：`**前 N-1 篇:**` 列表 + 统一收束金句

### 收束金句公式

```
把重复的事交给基础设施，把 {系列核心词} 留给 agent。
```

例：
- Vol.06: 把重复的事交给基础设施，把**创造力**留给 agent。
- Vol.07: 把重复的事交给基础设施，把**判断力**留给 agent。

### 互引规范

- 引用同系列其他文章时用 `[[wikilink]]` 格式（编译时转为脚注链接）
- 引用必须准确：序号、方法名、步骤数与其他文章一致
- 不确定时先查原文再引用

## 扩写方案选择决策表（2026-06-25 三种路径实测对比）

当首版中文字符低于目标 3000 时，从三种扩写方案中选择：

| 方案 | 操作 | 实测效果 | 适用场景 | 不适用场景 |
|------|------|----------|----------|------------|
| **A. 手动 patch 扩写** | 用 `patch` 工具在文件里插入 2-3 段新主题段落（每段 150-200 字），每插一段跑一次禁用词扫描 | 1646 字 → 3222 字，2 轮；不丢失结构；禁用词可控 | 单篇扩写、首次扩写、对质量敏感、需要保持原文案例一致性 | 需要批量扩写多篇文章、需要让模型基于原文重新组织结构 |
| **B. DeepSeek 二次扩写** | 把首版原文 + "扩写这 5 个章节到 X 字" 的指令作为新 prompt | 易丢失 markers（中文字符从 2092 → 2791，但 ===TITLE=== 被吃掉）、会再次触发「不是 X 而是 Y」、结构有时反而简化 | 大规模扩写、需要模型重新组织叙事线 | 时间紧迫、对 markers 完整性要求高（要继续发到飞书） |
| **C. 重置 prompt 让 DeepSeek 完整重写** | 第一次就写明确「中文字符 3200-3800 + 30+ h3 + 包含这些章节」，让模型一次性输出 | 一次性命中，但首版仍可能低于 3000 字（实测 2536、2075 两次都不达标） | 全新写作、首次发起 | 已经写完想补救 |

**决策规则**：
1. 首版字数 < 2500 → **首选方案 A**（手动 patch，最可控）
2. 首版字数 2500-3000 → 试方案 B 一次，如果 markers 丢了立即切方案 A
3. 一次性新写 → 用方案 C，但 prompt 必须显式给字数区间 + h3 数量 + 段长区间三个指标

**方案 A 的实操步骤**（手动 patch 扩写）：
1. `banned_scan.py` 扫当前文件，记录禁用词位置
2. 选 2-3 个扩写位置（推荐：① 第一个 h3 后加一段背景 ② 某个踩坑节后加一段延伸场景 ③ 结尾速查表后加一段「方法论本质」）
3. 用 `patch` 工具插入新段落，每段 150-200 字，含独立 ### h3 标题
4. 每插一段立即跑禁用词扫描
5. 最终扫一次确认全文件无禁用词、中文字数达标

2. **如果 `web_extract` 不可用**（工具可能在某些环境下缺失），直接用 `browser_navigate` 打开 URL，然后用 `browser_console` DOM 提取补全。实测 `web_extract` 在 2026-06 session 中不存在——不要等它报错再换方案，直接走 browser 路线。
3. **如果 `web_extract` 截断**（长文章常见），用 `browser_console` DOM 提取补全：
   ```javascript
   // browser_navigate 打开页面后，在 browser_console 里跑：
   (() => {
     const article = document.querySelector('article') || document.querySelector('main') || document.body;
     const result = [];
     const walk = (node) => {
       if (node.nodeType === 3) { const t = node.textContent.trim(); if (t) result.push(t); return; }
       if (node.nodeType !== 1) return;
       const tag = node.tagName.toLowerCase();
       if (tag === 'h2') result.push('\n## ' + node.textContent.trim());
       else if (tag === 'h3') result.push('\n### ' + node.textContent.trim());
       else if (tag === 'pre') result.push('\n```\n' + node.textContent.trim() + '\n```');
       else if (tag === 'p') result.push(node.textContent.trim());
       else if (tag === 'ul' || tag === 'ol') {
         node.querySelectorAll(':scope > li').forEach(li => result.push('- ' + li.textContent.trim()));
       } else if (tag === 'blockquote') result.push('> ' + node.textContent.trim());
       else { for (const child of node.childNodes) walk(child); }
     };
     walk(article);
     return result.join('\n\n');
   })()
   ```
   这个 IIFE 遍历 DOM 树，按标签类型映射到 markdown，能拿到完整正文（含代码块、列表、引用）。

### 质检脚本（每篇必跑，0 容忍）

```python
import re

# 完整 BANNED_PATTERNS（含空洞修饰词）。用户 2026-06-15 反馈：
# 只扫 AI 句式 + 弱化语气会漏掉标题里的"不是 X" 省略变体；
# 不含 空洞修饰会漏掉「非常/十分/极其/相当」。
BANNED_PATTERNS = [
    (r'不是[^，。]+[，,]而是', '不是X，而是Y'),
    (r'与其[说]?(?:[^，。]+)[，,]不[如说]', '与其X，不如Y'),
    (r'首先.*其次.*最后', '首先...其次...最后'),
    (r'值得注意的是', '值得注意'),
    (r'总的来说', '总的来说'), (r'总而言之', '总而言之'), (r'综上所述', '综上所述'),
    (r'在当今', '在当今'), (r'不可否认', '不可否认'),
    (r'众所周知', '众所周知'), (r'显而易见', '显而易见'),
    (r'不难发现', '不难发现'), (r'不言而喻', '不言而喻'),
    (r'毋庸置疑', '毋庸置疑'), (r'毫无疑问', '毫无疑问'),
    (r'正如我们所知', '正如我们所知'), (r'需要指出的是', '需要指出的是'),
    (r'(?<!\w)大概(?!\w)', '大概'), (r'(?<!\w)似乎(?!\w)', '似乎'),
    (r'一定程度上', '一定程度上'), (r'或多或少', '或多或少'),
    # 空洞修饰（必须加入，否则漏检）
    (r'(?<!\w)非常(?!\w)', '非常'), (r'(?<!\w)十分(?!\w)', '十分'),
    (r'(?<!\w)极其(?!\w)', '极其'), (r'(?<!\w)相当(?!\w)', '相当'),
]

# 扫描范围必须覆盖「全文件」，不能只扫 ===CONTENT=== 段。
# 用户 2026-06-15 实测：标题里"把 X 变成 Y，不是 Z"漏扫（因为只读了 body），
# 留下"不是 Z"省略变体未被处理。修正：直接扫整个 DeepSeek 输出文件。
with open('article.md', 'r') as f:
    content = f.read()

# 2026-06-20 实测：代码块内的 prompt 示例（如「确保问题不是显而易见的」「要非常深入」）
# 会被误判为禁用词。扫描前先剥离 ``` 代码块。
code_free = re.sub(r'```[\s\S]*?```', '', content)

hits = []
for pattern, label in BANNED_PATTERNS:
    matches = re.findall(pattern, code_free)
    if matches:
        hits.append((label, matches))

if hits:
    for label, matches in hits:
        print(f"❌ {label}: {matches}")
else:
    print("✅ 禁用词扫描通过（覆盖标题 + 副标题 + 正文）")

chinese = len(re.findall(r'[\u4e00-\u9fff]', content))
print(f"中文字符: {chinese} | 总字符: {len(content)} | 链接: {len(re.findall(r'https?://[^\\s)]+', content))}")
```

> ⚠️ **扫描范围 bug（2026-06-15 实测）**：旧版脚本只读 `===CONTENT===` 段，**漏掉了标题区的违规**（如"把 X 变成 Y，不是 Z"）。这是隐性 bug，因为正文扫描通过、但标题里仍有禁用结构。修正：扫整个 DeepSeek 输出文件（包含 `===TITLE===` 区）。

## 内容盘点（回答「写了哪些文章」时参考）

当用户问「有哪些成体系的文章」或「系列写到哪了」，先读 `references/content-pipeline-architecture.md`，它记录了 4 个存储目录、已知系列清单、搜索策略和 Pitfalls。不要做盲目的多目录搜索。

## Pitfalls

- **代码密集型文章的字数指标陷阱**（2026-06-15 实测）：质检清单中的「字数 2500-4500」指的是**中文字符数**，但代码密集型文章（15+ 代码块）的中文字符天然偏低——实测一篇 8970 总字符的文章仅 1716 中文字符。DeepSeek system prompt 中写「3500-4500 中文字符」时，模型倾向于用冗长中文描述补字数，反而降低质量。**修正方案**：DeepSeek prompt 改用「正文总字符 8000-12000（含代码），中文叙述不低于 1500 字」，并在质检时同时检查总字符和中文字符两个指标。总字符 > 6000 且代码块 ≥ 8 个时，中文字符 1500+ 即达标。
- **字数不足的扩写策略（2026-06-19 确认：patch 手动扩写 > DeepSeek 二次调用）**：直接创作（非 DeepSeek API）的初稿经常偏短。扩写优先级：① 加深技术细节（多给一个代码示例、多解释一层原理）② 增加对比表（和竞品/替代方案的对比）③ 补踩坑和边界条件 ④ 加实战场景。扩写后必须重跑禁用词扫描。
  
  **手动 patch 扩写法**（2026-06-19 实测首选）：DeepSeek 首版 1646 中文字符（目标 3200），用 `skill_manage(action='patch')` 直接在文章文件里插入 2-3 个新主题段落（每个 150-200 字），每插入一段后重新跑禁用词扫描。这种方法比 DeepSeek 二次调用更可控——不会丢失结构化 markers、不会越生成越短、不会引入新的禁用词变体（实测 DeepSeek 扩写时几乎必然产出「不是 X，而是 Y」句式）。典型扩写段落类型：① 与已有竞品/标准对比（如 "UI Skills vs ESLint"）② 个人踩坑案例 ③ 技术哲学层面的提炼 ④ 对团队/组织的实操建议。
- **DeepSeek 首次生成偏短需二次扩写**：代码密集型主题（Agent Skills、框架介绍等）首次生成的中文字符常在 1200-1500，低于 3000 目标。对策：在首轮 prompt 中就明确「每个 skill/概念必须配完整代码示例，不要只写文字描述」，避免二次 API 调用浪费时间。如果首版总字符已 > 6000 且结构完整，优先检查代码密度而非追加中文。
- **⚠️ DeepSeek 二次扩写会丢失结构化 markers（2026-06-18 实测）**：当首版字数不够，需要二次 prompt 让 DeepSeek「扩写 4 个章节 + 加 1 个新章节」时，**DeepSeek 会把首版输出的 ===TITLE=== / ===ALT_TITLES=== / ===COVER_PROMPT=== 这些 markers 全丢掉**，只输出扩写后的 CONTENT。**对策**：
  - 第一次 prompt 就把字数要求写明确（中文字符 3200-3800），避免二次扩写
  - 如果必须二次扩写：二次 prompt 让 DeepSeek 输出**完整重写版**（包含 markers），或在 python 里手动从原版拼接 markers + 新 content
  - 拼接示例（实测有效）：
    ```python
    import re
    raw = open('/tmp/xcode27_raw.md').read()
    expand = open('/tmp/xcode27_expand.md').read()
    title = re.search(r'===TITLE===\n(.+?)\n', raw).group(1)
    alt = re.search(r'===ALT_TITLES===\n(.+?)(?=\n===COVER)', raw, re.S).group(1)
    cover = re.search(r'===COVER_PROMPT===\n(.+?)(?=\n===CONTENT)', raw, re.S).group(1)
    final = f"===TITLE===\n{title.strip()}\n===ALT_TITLES===\n{alt.strip()}\n===COVER_PROMPT===\n{cover.strip()}\n===CONTENT===\n{expand.strip()}\n===END===\n"
    ```
- **❗ 段落分割器会携带残余 marker 进 segment 1（2026-06-25 实测）**：当手工扩写文章时，如果扩写过程把 `===TITLE===` 这类 marker 文本意外塞进了 CONTENT 段（典型原因：用 patch 工具在文件首行插入扩写段落时，old_string 没把 marker 行包括干净），paragraph splitter 按 `\n\n ` 切分时会把 `===TITLE===\n\n### 第一个 h3` 整体当作第一个段落，发到飞书后读者会看到 `===TITLE===` 字样。**修复 split 脚本**（实测有效）：
  ```python
  # 在 split 之前先剥离所有 marker 残留
  content = re.sub(r'===[A-Z_]+===\s*', '', content)
  content = content.replace('---', '')
  # 然后再按段落切分
  paragraphs = [p.strip() for p in content.split('\n\n') if p.strip()]
  ```
  同时检查 split 后的 segment[0] 不应以 `===` 开头——这是 marker 泄漏的快速诊断信号。
- **lark-cli post 消息 JSON 作为 CLI 参数会失败（2026-06-23 实测）**：`lark-cli im +messages-send --msg-type post --content <json>` 当 JSON 内容超过 200-300 字符时，shell 参数截断导致 lark-cli 报 `Usage:` 而不是真正的错误。不是 lark-cli bug，是 shell ARG_MAX 限制。**对策**：用 Python `subprocess.run(...)` 通过 stdin 传 JSON，或写临时文件后 pipe。`--markdown` 分段发送不受影响（每段 < 900 字符）。已更新到 `wechat-cover-html` 和 `tech-content-writer` 两个 skill。
- **❗ patch 插入新段落时 h3 标题被正文吞掉（2026-06-28 实测，apple/container 案例）**：手动 patch 扩写时，如果 `old_string` 选的是「前一段正文」而 `new_string` 里只有 `### 新标题` 没有显式空行分隔，patch 工具会把新标题直接拼到前一段末尾，变成 `…前一段。### 新标题` 一个超长行。这不会立即报错，但后续按 `\\n\\n` 切段时不会把它识别成新段，整段被当成前段的延续。**debug 信号**：用 `grep -n '^### ' file.md` 看所有 h3 标题行，如果某个标题行特别长（如 80+ 字符），说明它被吞了。**修复**：再 patch 一次把 h3 标题行独立成行（确保 `### xxx` 前后都有 `\\n\\n` 包裹）。**预防**：patch 时 new_string 写成形如 `\\n\\n### 新标题\\n\\n新段落正文…`，前导和尾随的 `\\n\\n` 都要显式写出来。

## 扩写决策表二次验证（2026-06-28 apple/container 案例）

继 2026-06-25 决策表建立后，2026-06-28 Apple Container 推文文章再次验证方案 A 有效：

- **首版**：1944 中文字符 / 19 h3 / 段长中位数 44（段太碎 + 字数太少）
- **6 处 patch 扩写**（未来场景扩段、builder 调优、注册表凭据踩坑、技术栈对比维度、个人经历）→ **3035 中文字符 / 21 h3 / 段长均值 57**
- **禁用词扫描**：0 命中（patch 时每插一段跑一次）
- **总耗时**：扩写环节约 5 分钟（6 次 patch + 6 次扫描），比 DeepSeek 二次扩写（实测 3-5 分钟但会丢 markers）更稳

**新增的关键经验**：
1. **短篇「探索型」文章（推文→解读）首版字数普遍 1500-2000**，比模板 A「工具介绍」更短（后者首版常 2500-3000）。**所有探索型文章**都该默认走方案 A。
2. **patch 扩写时优先扩「未来场景 / 实操经验」**——这两类内容每段都自然有 150-200 字可写，且符合 AIGC 合规的「增量信息」要求。
3. **段长中位数低于 50 的初稿**，patch 时不要扩旧段（会把短段变中段但整体结构不变化），优先**加新段**（每段独立 h3），这样能同时拉高 h3 数和段长中位数。
- **DeepSeek 二次生成反而比首次更短（2026-06-18 实测反直觉）**：首版 2536 中文字符 + 31 h3，二次 prompt 加了"扩到 3200+"的指令，**反而生成 2075 字 + 20 h3**。模型倾向"压缩"而不是"扩写"。**对策**：第一次 prompt 就要给具体数字（3200-3800 + 25+ h3），别赌"再来一次会更好"。
- **「不是 X，而是 Y」是最高优先级禁用**：这个句式是 AI 写作最典型的标志，必须在 prompt 和 post-check 两个层面拦截。即使 DeepSeek system prompt 中列了禁用，仍可能漏网——post-check 扫描不可省略。
- **官方文档链接必须验证可访问**：写完后用 curl HEAD 检查每个链接的 HTTP 状态码。死链比没链接更糟糕。
- **代码示例不要省略上下文**：`import` 语句、环境要求、版本号必须写明。读者复制粘贴后跑不通 = 文章失败。
- **技术文章的「由浅入深」不等于「啰嗦」**：铺垫是为了降低理解门槛，不是凑字数。每段铺垫后面必须跟实质内容。
- **系列文章的收束金句必须与系列主线一致**：每篇的金句核心词不同，但句式统一。写之前先看前几篇的金句。
- **DeepSeek temperature 比育儿号低**：技术文更注重准确性（0.7），育儿号更注重文采（0.8）。不要搞混。
- **避免「教程体」**：技术公众号 ≠ 官方文档翻译。要有观点、有踩坑、有「我在项目中的做法」。纯教程体读者直接去看文档。
- **delegate_task context 不能太短**：子 agent 没有对话历史，风格规则、禁用词、代码示例必须完整传入 context 字段。
- **Python f-string 与 JS 模板字符串冲突（2026-06-19 实测）**：当 USER_PROMPT 包含 JavaScript 代码（如 FLIP 技术的 `${first.left - last.left}`），如果用 f-string 构建 prompt，Python 会把 `${...}` 尖括号内的内容当表达式执行报错。对策：① USER_PROMPT 用普通字符串拼接（不用 f-string）② 或用 `${'${first.left - last.left}'}` 嵌套转义 ③ 最佳实践：把代码示例直接写死在字符串中，不要用变量插值。详见 `references/deepseek-api-script-template.md` 的 Common Failure Modes 表。
- **用户说「继续」「继续完成剩下的」= 不要问确认**：写完一篇直接下一篇，不要停顿询问是否继续。用户说「先发布第一篇」= 直接执行完整发布流程，不要问「确认发布？」。
- **❗ 发布流程是原子任务，不要分步问（用户 2026-06-29 两次强纠偏）**：用户说「发布到公众号」/「发布到 AICoder」/「整理并发布」时，**整个发布链路是一个 task**，不要在「要不要生成封面」「要不要发摘要」等环节停下来问。具体表现：
  - 用户原话 1：「**你为什么没有生成封面？**」（你已经发完摘要 + 正文，跳过了封面）
  - 用户原话 2：「**为什么老是问我**」（你在写完后连续问了「是否继续」「是否需要封面」）
  - 修正：完整发布链路 = 写文章 + 禁用词扫 + 生成封面 + 上传 OSS + 发封面图链接 + 发标题摘要 + 分段发正文。每一步都直接执行，全部完成后再汇报。**唯一允许中断的场景是用户主动叫停或外部依赖不可用**（如 OSS 上传凭证缺失、lark-cli 调用被系统阻断）。
- **执行流程清单（一次性完成，不要在中间问）**：
  1. 写文章 → 立即禁用词扫描 → 修到 0 命中
  2. 生成封面 HTML → 渲染 PNG → 上传 OSS → 得到 URL
  3. 发标题摘要 + 封面 URL（一条 markdown 消息）
  4. 分段发正文（每段 ≤ 880 字符）
  5. 一句话汇报：标题 + 封面 URL + 段落数 + 已发到哪个 chat
- **执行完后再问用户**：如果用户对结果不满意，**用户会主动叫停**。不要预防性地问「这样可以吗」。
- **用户说「直接补」「你为什么没有生成封面」「为什么老是问我」= 发布流程还没走完，立刻补齐，不要再提问**（2026-06-29 实测强纠偏）。当用户已经要求发布、而封面 / 摘要 / 封面链接明显缺失时，正确行为是直接生成封面、上传 OSS、补发链接，随后用一句话汇报结果。
- **飞书分段按段落切，不按字数切**：用 `\n\n` 边界分割，最大 880 chars/段（900 限制留 20 余量）。`---` 分隔符必须删除（飞书渲染为水平线破坏格式）。
- **lark-cli 发送命令的正确写法**：`lark-cli im +messages-send --as user --chat-id oc_xxx --markdown "..."`。注意是 `im` 不是 `+im`（lark-cli 把 `im` 当成 service 名而不是子命令）。`--markdown` flag 自动 wrap 成 post 格式并处理图片 URL。`--text` 是纯文本。常用 flag 速记：`--as user`（个人 OAuth 身份）、`--chat-id`（群聊）、`--user-id`（私聊）、`--idempotency-key`（防重复）。
- **「与其...不如」是禁用句式**：这个变体容易漏检。修复方式：把「与其X，不如Y」改写为直接陈述句。例：「与其等升级，不如现在处理」→「升级那天集中修成本很高，现在处理更合理」
- **英文→中文改编是「不是X，而是Y」的重灾区（2026-06-15/16 实测，连续 3 篇推文改编均触发）**：英文对比结构（"not X, but Y"、"X is not about Y, it's about Z"）翻译成中文时几乎本能地映射为「不是X，而是Y」。实测一篇 2700 字的改编文章初稿出现 7 处该句式，需要 2 轮修复。对策：写完后立即跑禁用词扫描，不要等全部写完再统一检查——边写边扫描效率更高。
- **空洞修饰词需加入禁用词扫描**：「非常」「十分」「极其」「相当」在扫描脚本的 BANNED_PATTERNS 中容易被遗漏（主列表只有弱化语气词）。建议在扫描脚本中统一加入：`r'非常', r'十分', r'极其', r'相当'`。本 SKILL 的「空洞修饰」清单列了这些词，但扫描脚本没覆盖——下次使用时同步更新脚本。
- **❗ 禁用词扫描必须覆盖整篇文章，不能只扫 ===CONTENT=== 段**（2026-06-15 实测）：DeepSeek 输出通常分 `===TITLE===` / `===COVER_PROMPT===` / `===CONTENT===` 三段。旧脚本用 `re.search(r'===CONTENT===...')(.*?)===END===` 抽 body，**导致标题区的"不是 X"省略变体漏检**（如"把 X 变成 Y，不是 Z"）。修复：扫整个 DeepSeek 输出文件，扫描后再人工挑出 title 区违规。
- **❗ AIGC 二次创作必须 fact-check，不能"为了差异化"瞎编细节**（2026-06-15 实测）：把英文推文扩写为中文文章时，容易"为了让故事更生动"加新事实，结果写错。例如原文说"Bun 从 Zig 移植到 Rust"，扩写时改成"Zig → C"，发布前**没核对原文**就发了。**对策**：扩写后、发布前对**所有技术细节**做一遍 fact-check（语言名/数字/版本号/人名/引文）。重点检查"为了差异化"加的"新场景"是否还忠于事实——可以加观点、加个人案例，但**不能改原文的技术数据**。
- **推文→文章的禁用词扫描要尽早**：实测第一篇写了 2700 字才发现 4 处「不是X，而是Y」，修完后字数又不够。建议在写完第一个完整章节后就跑一次扫描，及时修正方向，避免全文写完再返工。
- **推文→文章转换的中文字数陷阱（2026-06-15/16 实测）**：推文→中文技术文章的初稿中文字数普遍偏低（实测 3 篇分别为 1871、1603、1113，目标 2500）。原因：推文本身是英文，转写时大量英文术语、代码块、链接、技术名词不计入中文字数，但占据了大量篇幅。**对策：初稿就写 3500+ 中文字的纯中文叙述量**，不要等到质检后才开始扩写。具体做法：每个章节至少 300 字纯中文分析，技术概念用中文解释而非直接贴英文原文，案例/场景用中文展开而非一行带过。
- **推文改编文章「不是X，而是Y」几乎必然出现（2026-06-15/18 连续 4 篇实测）**：每篇初稿都触发 2-4 处该句式。英文对比结构（"not X, but Y"、"X is not about Y, it's about Z"）翻译成中文时自动映射为该句式。**对策**：写完第一个完整章节后立即跑禁用词扫描，修完再继续写下一章。不要等全文写完再统一检查——全文 2500 字 + 4 处禁用词 = 需要全文 patch 4 次，效率极低；边写边扫描每次只需 patch 当前章节。
- **直接创作（无 DeepSeek）的初稿中文字数偏低（2026-06-20 实测，连续 5 篇）**：不走 DeepSeek API、直接在 write_file 里写文章时，初稿中文字符普遍在 1100-1800（目标 2500）。原因：直接写作倾向于精炼，每个章节只写核心观点而不展开分析。**对策**：初稿写完后立即跑字数检查，如果 < 2500，在 2-3 个位置插入扩展段落（每段 150-200 字），类型包括：① 案例展开 ② 与其他方案的对比分析 ③ 行业趋势延伸。每插一段立即跑禁用词扫描。实测 4-6 轮 patch 可从 1500 扩到 2500。
- **禁用词扫描在代码块内产生误报（2026-06-20 实测）**：当文章包含 prompt 示例代码块（如「确保问题不是显而易见的」「要非常深入」），扫描脚本会把代码块内的文字也当作正文扫描，产生误报。**修复**：扫描前用 re.sub 剥离 ``` 代码块再扫描。已更新到脚本中。
- **🧵 emoji 不等于 thread 存在**：用户说"获取全部 thread"时，fxtwitter 返回的 `tweet.thread` 字段可能空，但 `tweet.text` 含 `🧵` —— 作者只是预告要开 thread，实际还没发。**先判定 thread 是否真实存在**：(1) `tweet.thread` 数组 `len > 1` 才算真 thread (2) `in_reply_to_status` 存在是回复推文 (3) `browser_navigate` 打开推文 URL 看 snapshot 里的 "Read N replies" 是 thread 长度的最准参考。确认"只有 1 张图 + 1 句话"后直接用图 + 官方资料扩写，不要再去找不存在的 thread。实测：v_pradeilles "What's new in Xcode 27" 这条只有 1 张图、Views 仅 1587。
- **同一素材写 v1 + v2 时强制差异化（2026-06-18 WWDC26 Xcode 27 实测）**：用户重复要求"收录"同一推文/主题时，不要扩写 v1（= 降权），直接发 v2，**第二个完全不同的角度**。AIGC 合规要求"增量信息"：(1) DeepSeek prompt 显式列 v1 已覆盖的章节标题，禁止重复 (2) 6 段主结构必须重新设计 (3) 真实案例必须新（不能复用 v1 的同一案例 5 次）(4) 文末对比段写明"v1 写了 X，v2 写在 v1 视角之外你需要立刻改的事" (5) OSS 资源 key 加 `-v2` 后缀避免覆盖 (6) 飞书第 1 段标注"v2 段落化重写版"。
- **DeepSeek 图片占位符的隐性 bug（2026-06-18 实测）**：DeepSeek 偶尔返回 `![alt 描述文字](无 URL)` 这种"空括号图片语法"，alt 是描述性文字而不是图片名，导致 `re.findall(r'!\[[^\]]*\]\([^)]+\)', body)` 漏数。**修正**：扫描时用更宽松的 `r'!\[[^\]]*\]'`；或在 prompt 里强制"图片必须用完整 `![alt](https://...)` 语法，不能用空括号"。同时**手动核对首段图链接**——如果扫到 0 张图但 prompt 要求"开场 1 张图"，优先排查这种 bug 而不是直接补图。
- **Twitter Article 的 `tweet.text` 为空**：长文（draft.js 格式）抓取后 `tweet.text` / `tweet.raw_text.text` 都是空字符串，正文在 `tweet.article.content.blocks[]` 里。block type 映射：`header-two` → `## `、`header-three` → `### `、`unstyled` → 段落、`unordered-list-item` → `- `、`ordered-list-item` → `1. `、`atomic`（图片/嵌入）→ 跳过。详见 `references/x-twitter-capture.md`。
- **推文含多个外链文章的综合写法**：推文本身可能只是 3 个链接 + 简短描述（如"@twannl 分享了 3 篇 SwiftLee 文章"），需要逐个 `web_extract` 每个链接获取完整内容，再综合为一篇公众号文章。写文章时选一个主线角度（如"Xcode 27 Agent Skills"），其余文章作为补充素材（如第三方 Skill、async/await 迁移）。不要写成 3 篇独立文章的拼接，要有统一的叙事线。
