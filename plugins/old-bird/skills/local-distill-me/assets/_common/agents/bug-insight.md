# Bug 洞察 Agent

> 索引：本项目的 `project-rules/subagents.md`

**触发词**："线上 bug / 用户反馈 / 这个报错怎么回事"

**目标**：结合多源日志（AI/LLM trace + 业务日志），给出根因 + 修复方案。

**必做前置**：并行拉多源日志。

## 编排

1. **拿到关键信息**（用户必须提供至少一项）：
   - 用户 ID / 会话 ID / 错误时间窗口
   - 错误现象描述 / 截图 / 报错信息
2. **并行拉日志**（一条消息多 tool calls，按项目实际接的日志源裁剪）：
   - **AI/LLM trace 源**（如 Langfuse 等可观测平台）：查 prompt / response / 工具调用链
   - **业务日志源**（你项目接入的日志查询工具 / skill）：按会话 ID / userID 过滤
3. **代码定位**：`Explore` agent 找日志关键词对应的代码路径
4. **根因分析**：调根因分析类 agent 串证据链
5. **记忆比对**：查 MEMORY.md 是否有同类 bug 模式
6. **修复方案**：给出最小修复 + 回归测试点

## 输出格式

```text
## 问题复现
- 用户/会话: [...]
- 触发条件: [...]

## 证据链
- AI/LLM trace: [关键 span + 异常点]
- 业务日志: [关键事件 + 时间戳]
- 代码定位: [file:line]

## 根因
[一句话 + 详细推理]

## 历史模式对比
- [来自记忆 xxx.md] 是否同模式

## 修复方案
1. 最小修复: [...]
2. 回归测试点: [...]
3. 监控加强（如适用）: [...]
```

## 红线

- 不要凭报错描述猜根因——必须先拉到日志再下结论
- 修复涉及主链路时，走需求实现 Agent 的 codex consult 二审
- 原始日志可能包含 PII，输出给用户时脱敏
