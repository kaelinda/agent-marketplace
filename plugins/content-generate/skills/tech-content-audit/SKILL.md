# 技术内容审核 SKILL (tech-content-audit)

## 定位

技术公众号文章发布前的**最后一道质量门控**。从读者视角出发，判断文章是否值得发布。核心原则：**技术准确性第一，读者能用、能懂、能收益**。

## 五大审核维度

### 一、技术准确性 ✅（最高优先级）

| 检查项 | 说明 |
|--------|------|
| 代码可运行性 | 代码示例是否可执行？是否缺少 import、环境变量、版本依赖？ |
| 技术概念正确性 | 概念解释是否准确？有无误导性表述？ |
| API/工具用法 | 调用的 API 是否存在？参数是否正确？返回值描述是否准确？ |
| 版本号标注 | 是否标注了适用的版本范围？有没有使用已过时的 API 而未标注？ |
| 官方文档一致性 | 关键技术点是否与官方文档一致？有无臆测或过时的信息？ |

**判断标准**：

| 信号 | 判定 |
|------|------|
| ✅ 正确 | 代码可运行 + 概念准确 + 版本已标注 |
| ⚠️ 需标注 | 技术正确但用了旧版 API 未标注、代码缺少前置条件说明 |
| ❌ 错误 | 未验证的代码片段、错误的 API 调用、概念性错误 |

### 二、实用性 ✅

| 检查项 | 说明 |
|--------|------|
| 可操作性 | 读者能否照着文章完成一个实际任务？ |
| 代码示例完整性 | 是否提供了完整的、可复制的代码示例？还是只给了 API 名？ |
| 实际应用场景 | 是否给出了真实场景而非纯理论？ |
| 步骤清晰度 | 是否有清晰的步骤说明，读者不会迷路？ |

**判断标准**：

| 信号 | 判定 |
|------|------|
| ✅ 符合标准 | 可复制的教程 + 完整代码 + 真实场景 |
| ⚠️ 需补充 | 纯理论探讨（需补实践案例）、只有概念解释（需加代码示例） |
| ❌ 不合格 | 只有抽象描述、没有可执行内容 |

### 三、深度 ✅

| 检查项 | 说明 |
|--------|------|
| 独到见解 | 是否有原创观点、个人实战经验？而非浅层翻译或文档搬运 |
| 原理层面 | 是否深入到"为什么这样设计"的层面，而非只停留在"怎么用"？ |
| 对比分析 | 是否有与替代方案的对比？有无取舍分析？ |
| 常见坑点 | 是否指出了读者容易踩的坑？ |

**判断标准**：

| 信号 | 判定 |
|------|------|
| ✅ 符合标准 | 深入原理 + 实战经验 + 独到观点 |
| ⚠️ 需补充见解 | 只罗列功能无分析、有代码但无"为什么这么写" |
| ❌ 不发布 | 外文翻译无原创观点、纯文档搬运、无增量信息 |

### 四、风险标注 ⚠️

以下情况**必须标注**，缺一不可：

| 风险类型 | 标注方式示例 |
|---------|-------------|
| 过时技术 | `"适用于 Python 3.8，新版已弃用"` |
| 实验性功能 | `"Beta 功能，生产环境谨慎使用"` |
| 第三方依赖 | `"依赖 X 服务，需自行部署"` |
| 安全风险 | `"示例代码仅供演示，生产需加固"` |
| 性能敏感 | `"大数据量下需调整参数，默认配置不适用生产"` |
| 兼容性限制 | `"仅支持 macOS/Linux，Windows 需额外配置"` |

**审核时逐个对照，发现上述任何一种情况但文章未标注 → ⚠️ 需补充**。

### 五、禁止内容 ❌

绝对不发布：

| 禁止项 | 说明 |
|--------|------|
| 未经验证的代码 | 作者自己都跑不通的代码 |
| 安全漏洞示例 | 明文密码、SQL 注入式示例、无鉴权的敏感操作 |
| 直接抄袭 | 未经改写的搬运内容（违反 AIGC 合规） |
| 技术无关营销 | 产品推销、引流文案、与技术主题无关的推广 |
| 事实性错误 | 数据/版本号/人名/引文与原文不一致 |

## 审核流程

### Step 1: 通读全文 → 理解技术主题

- 文章核心主题是什么？目标读者是谁？
- 文章承诺给读者什么价值？（教程？原理？决策框架？）

### Step 2: 检查代码 → 语法、逻辑、版本

- 逐个代码块检查：
  - 语法是否正确？
  - import 是否完整？
  - 变量是否定义？
  - 版本兼容性是否说明？
- 尝试用 execute_code 或 terminal 运行关键代码片段（当环境允许时）

### Step 3: 评估深度 → 是否有原创观点

- 是否有"作者作为开发者会怎么选"的判断？
- 是否有官方文档之外的工程经验？
- 是否只是"把文档翻译成中文"？

### Step 4: 标注风险 → 过时/实验性功能/安全

- 逐项对照"必须标注"表格
- 每一项未在文中标注的风险 → 记录为审核意见

### Step 5: 决策发布

| 结果 | 含义 | 后续动作 |
|------|------|---------|
| ✅ 通过 | 五大维度均达标 | 进入发布流程 |
| ⚠️ 需修改 | 有问题但可修复 | 给出具体修改清单，作者修正后复审 |
| ❌ 拒绝 | 存在不可接受的缺陷 | 说明拒绝原因，建议重写或放弃 |

## 审核报告模板

审核完成后，输出以下格式的报告：

```markdown
## 📋 技术内容审核报告

**文章标题**：{标题}
**审核时间**：{时间}

### 一、技术准确性
**评级**：✅ / ⚠️ / ❌
**问题**：
- [具体描述，如无问题写"未发现"]

### 二、实用性
**评级**：✅ / ⚠️ / ❌
**问题**：
- [具体描述]

### 三、深度
**评级**：✅ / ⚠️ / ❌
**问题**：
- [具体描述]

### 四、风险标注
**评级**：✅（无遗漏）/ ⚠️（有遗漏）
**遗漏项**：
- [列出应标注但未标注的风险，如无写"无遗漏"]

### 五、禁止内容
**评级**：✅（无禁止内容）/ ❌（发现禁止内容）
**发现**：
- [列出问题，如无写"未发现"]

---

### 🏁 审核结论
**结果**：✅ 通过 / ⚠️ 需修改 / ❌ 拒绝

**修改清单**（仅当 ⚠️ 时）：
1. [具体可执行的修改项]
2. [具体可执行的修改项]

**拒绝理由**（仅当 ❌ 时）：
- [具体不可接受的缺陷]
```

## 与发布流程的集成

### 自动审核触发点

在 `tech-content-writer` 写作完成后、`wechat-publisher` 发布前，自动执行本审核：

```
写作完成 → [tech-content-audit] → ✅ 通过 → 发布
                            → ⚠️ 修改 → 作者修正 → 复审
                            → ❌ 拒绝 → 终止发布，报告原因
```

### 与已有质检的关系

本审核是**补充**而非替代现有的 `tech-content-writer` 质检清单：

| 已有质检 | 审核补充 |
|---------|---------|
| 禁用词扫描（AI味检查） | **不负责**禁用词，由 `tech-content-writer` 处理 |
| 字数/段落检查 | **不负责**格式，由 `tech-content-writer` 处理 |
| AIGC fact-check | **重叠**但更聚焦技术事实（代码/版本/API） |
| 合规检查 | **重叠**但增加了实用性/深度/风险标注维度 |

**审核时不需要重复跑禁用词扫描**——那是写作 skill 的职责。审核聚焦"内容本身的质量"，而非"文风"。

## Pitfalls

- **审核不是重写**：审核者只指出问题，不代替作者修改。修改由作者（或 `tech-content-writer`）完成。
- **代码审核的边界**：如果文章代码依赖特定环境（如数据库、GPU、特定 OS），审核者应在报告中注明"代码在 X 环境下未验证"，而不是直接判定为 ❌。
- **区分"过时"与"历史"**：如果文章的主题就是"某个旧版本的历史用法"（如"Python 2 到 3 迁移指南"），使用过时 API 是合理的，但必须标注版本适用范围。
- **实用性和深度的权衡**：入门教程偏重实用性，深度原理文章偏重深度——审核时根据文章定位调整权重，不要求每篇同时满分。
- **审核报告的语气**：用客观描述指出问题，不用贬低性语言。"这段代码缺少异常处理"而非"这段代码写得烂"。
- **❌ 拒绝是最后手段**：优先给 ⚠️ 修改清单，只有以下情况才 ❌：事实性错误 + 无法修正（如核心技术原理错误）、完全抄袭、纯营销内容。
