# 段落化 + 甜点字数迭代 playbook

> 起源：2026-06-15 用户反馈「太短了，并且没有段落，请重写」后，三版迭代总结。
> 适用：所有非代码密集型的中文技术公众号文章（行业解读 / 观点类 / 实操指南）。

## 用户原话与意图

| 用户说法 | 实际意图 |
|---|---|
| 「没有段落」/「段落化」 | 每段独立 h3 标题 + 段间空行 + 段长可控 |
| 「太短」 | 中文字符未命中 3000-4000 甜点区 |
| 「请重写」 | 重新生成（不是 patch 现有） |

**关键**：用户用「段落化」时，**段长和字数是双向约束**。如果只满足其中一个（如"30 段但每段 30 字"= 900 字总长），用户还会再要求一次重写。

## 目标区间

| 指标 | 目标值 | 区间 |
|---|---|---|
| 中文字符 | 3200-3700 | 2500-4500 可接受 |
| 段数 | 30-40 段 | 25-50 段 |
| 段长中位数 | 100-200 字 | 80-250 字 |
| h3 标题数 | 25-35 个 | 20+ |
| 图片 | 1+ 张（如有原图） | 0-3 张 |
| 链接 | 4-6 个 | 2+ 必填 |

## DeepSeek Prompt 模板（段落化专用）

```python
prompt = """
# 段落结构（用户硬要求）
- 每段 100-200 字
- 每段独立 h3 标题（### xxx）
- 段与段空行隔开
- 列表用 - 或 1.，每条 bullet 独立短句
- 资源区每条链接带一行简介

# 三个数字必须同时命中
- 总字数：3200-3700 中文字
- 段数：30-40 段
- h3 标题数：25+ 个

# 当字数不够时扩写策略（按优先级）
1. 每段多说一层「为什么是这样」+「我第一次遇到这事的场景」
2. 在合适位置新增独立小段（如「verifier 维护的三个现实负担」独立成 3 段）
3. 资源区扩成 6 个独立小段（每段：标题 + 一行简介 + 链接）
4. 加「附录：N 种反模式」+ 每条独立 h3 段 + 一段总结

# 绝对禁止
- "不是 X，而是 Y" 及其变体
- 空洞修饰：非常 / 十分 / 极其 / 相当
- 弱化语气：大概 / 可能 / 应该 / 似乎
"""
```

## 三版迭代的真实案例（2026-06-15 Loop Engineering）

### v1：第一次生成

```
中文字符: 2720  ← 用户嫌短
段数: 12
h3: 4
段长中位数: 800+ ← 用户嫌"没段落"
诊断: 散文化（5-10 段 / 3000-4000 字）
```

### v2：尝试"段落化"

```
中文字符: 1997  ← 更短了
段数: 43
h3: 21
段长中位数: 47  ← 太碎
诊断: 段已密但字数上不去（DeepSeek 倾向"短段 + 少字"）
```

### v3：明确给三个数字

```
中文字符: 3016  ✅ 命中
段数: 36
h3: 34
段长中位数: 161  ✅ 100-200 区间
诊断: 同时满足
```

**v2 → v3 关键改动**：DeepSeek prompt 里**显式给三个具体数字**（字数 3000-4000、段数 30+、h3 数 25+），而 v2 只说"段落化"。

## 手动扩写 vs 重新生成

| 情况 | 策略 |
|---|---|
| 字数差 < 500 | 手动扩写（4-5 段加深度） |
| 字数差 500-1000 | 重生成 + 手动补一两段 |
| 字数差 > 1000 | 完整重生成，明确给三个数字 |
| 段已密但字数少 | 加新章节（如"反模式清单"或"踩坑扩展"） |
| 探索型文章首版普遍 1500-2000 字 | **直接走手动 patch 扩写**（2026-06-28 实测，apple/container 案例） |

**手动扩写最容易做的三件事**：
1. 在收束段前加一段"具体怎么做"（3 步行动 + 工具/链接）
2. 资源区从 5 个链接扩到 6 个独立小段
3. 附录加 5 条反模式，每条独立 h3 段

**2026-06-28 探索型文章 patch 扩写首选段位**（实测扩得动、不引入禁用词、符合 AIGC 合规）：
- 「未来场景之 N」每段扩写一层"具体场景"（150-180 字）
- 「个人实操经验」加 1-2 个新踩坑
- 「与 X 对比」段落插一个"对比维度补充"
- 资源区从 3 个链接扩到 4 个独立小段

**实测数据**（2026-06-28 apple/container）：
- 首版 1944 中文字符 / 19 h3 → 6 处 patch 后 3035 中文字符 / 21 h3
- 段长均值 47 → 57（中长段主导，但仍有部分短段拉低中位数）
- 总耗时 5 分钟（6 次 patch + 6 次禁用词扫描）
- 决策表规则命中：首版字数 < 2500 → 首选方案 A ✅

## 段落结构扫描脚本

```python
import re

def check_structure(body):
    """检查段落化 + 甜点字数结构。返回命中状态。"""
    paragraphs = [p.strip() for p in body.split("\n\n") if p.strip()]
    chinese = len(re.findall(r'[\u4e00-\u9fff]', body))
    h3s = re.findall(r'^### .+$', body, re.MULTILINE)
    lens = [len(p) for p in paragraphs]
    median_len = sorted(lens)[len(lens)//2] if lens else 0

    result = {
        'chinese_chars': chinese,
        'paragraph_count': len(paragraphs),
        'h3_count': len(h3s),
        'median_paragraph_len': median_len,
        'in_sweet_spot': (
            3000 <= chinese <= 4000 and
            25 <= len(paragraphs) <= 50 and
            80 <= median_len <= 250
        ),
    }
    return result

# 命中区间判定
# 3000-4000 字 + 25-50 段 + 段长 80-250 = 甜点区
# 任何一项不满足 = 需要重写
```

## 段落化与文章结构模板的对应

| 模板 | 段落化时 | 段数 | 字数 |
|---|---|---|---|
| 模板 A（工具/框架介绍） | 6 个章节 × 5-6 段 = 30-36 段 | 30-40 | 3000-4000 |
| 模板 B（问题排查） | 5 个步骤 × 6-7 段 = 30-35 段 | 30-35 | 2500-3500 |
| 模板 C（原理深挖） | 5 个层次 × 6-7 段 = 30-35 段 | 30-35 | 3000-4000 |
| 综合全景 | 6-8 个方向 × 5-6 段 = 30-48 段 | 30-50 | 4000-6000 |

**模板 A 在段落化时最容易命中甜点区**。模板 B 因为步骤紧凑，段数可能偏少，要补 5-6 个独立 h3（如"3 个常见误区"）。

## 关键避坑

1. **"段落化"不等于"碎段化"**：每段 100-200 字，不要每段 30-50 字。碎段读起来像大纲不像文章。
2. **"短"不是罪**：30 段 × 100 字 = 3000 字 = 甜点区。不要"为了凑 40 段"写空段。
3. **DeepSeek 不会自动平衡**：必须给三个数字（字数/段数/h3 数），否则它会偏向某一极端。
4. **段落化不与"代码块密集"冲突**：代码块独立成段不计入段数。技术文章代码块 + 文字 7:3 比例时，文字段 20-25 段 + 代码块 8-10 个 = 仍命中甜点区。
5. **收束段不需要 h3**：3000+ 字文章的收束段通常是 100-200 字的总结，**独立成段但不需要 h3 标题**。h3 数 ≠ 段数（h3 数 -1 至 h3 数 = 段数）。
