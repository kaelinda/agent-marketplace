# doc-templates — 7 份 onboarding 文档的骨架与内容要求

生成 `docs/onboarding/` 时逐份对照。骨架里的 `<...>` 是占位说明，不要原样输出；
章节可按项目实际增删，但「硬性要求」不能少。所有 frontmatter 字段
（title/order/icon/summary）必填 —— HTML 的侧边栏、卡片、排序都靠它。

---

## README.md（导读，order: 0，icon: 📖）

```markdown
---
title: 导读
order: 0
icon: 📖
summary: 5 分钟速览 + 按角色的阅读路线图
---

## 这套文档怎么读

> 一句话：<项目定位>。第一天先读 01 和 05，能跑起来再看 02/04。

| 你是谁 | 建议路线 |
|--------|---------|
| 第一天入职的新人 | 01 概览 → 05 快速上手 → 03 目录导览 |
| 要改业务逻辑 | 04 核心流程 → 03 速查表 → 06 术语表 |
| 要做架构评审 | 02 架构总览 → 04 核心流程 |

## 5 分钟速览

<5~8 个要点：项目是什么、核心模块、技术栈、怎么跑、去哪问>
```

## 通用规则

- 语言跟随仓库（中文项目→中文文档，英文项目→英文文档，术语表始终中英对照）。
- 长度手感：01/02/04 每份 60~150 行；README/06 可短。宁可精准简短，
  不要凑字数 —— 新人时间宝贵。
- mermaid 节点文字避免 `(` `)` `"` 与 `[` `]` 嵌套；`-->` 两侧不留空格也可以。
- 所有事实性描述（版本号、命令、路径）生成时逐一核实，不确定就标「待确认」。
